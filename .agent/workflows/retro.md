# 🔄 회고 및 개선 워크플로우 (Retrospective)

## 📋 개요

지난 작업을 돌아보고 개선점을 찾아 프로세스를 진화시키는 회고 회의입니다.
성공과 실패를 모두 분석하여 팀의 집단 지성을 문서화합니다.

---

## 🔄 실행 단계

### Phase 1: 준비 (Preparation)

#### Step 1.1: 데이터 수집

**수집할 정보**:

1. **완료된 작업**:
   - GitHub Closed Issues/PRs
   - 배포 로그 (`.agent/memory/DEPLOYMENT_LOG.md`)
   - 완료된 기능 목록

2. **메트릭**:
   - 평균 PR 리뷰 시간
   - 배포 빈도
   - 버그 발생 건수
   - 평균 해결 시간

3. **이벤트**:
   - 프로덕션 장애
   - 긴급 핫픽스
   - 기술 부채 누적

**자동 수집 스크립트** (선택):

```bash
# 지난 한 달간 머지된 PR 개수
gh pr list --state merged --search "merged:>2026-01-01" --json number,title | jq length

# 지난 한 달간 발생한 이슈 개수
gh issue list --state closed --search "closed:>2026-01-01" --json number,title | jq length
```

### Phase 2: 회고 회의 (Retrospective Meeting)

#### Step 2.1: 체크인 (5분)

**목적**: 참석자들의 현재 기분과 상태 파악

**질문**:

- "지난 한 달 작업을 한 단어로 표현한다면?"
- "에너지 레벨은? (1-10)"

**규칙**:

- 비난 금지
- 솔직하되 건설적으로
- 모든 의견은 존중

---

#### Step 2.2: 잘한 점 수집 (15분)

**방식**: 각자 포스트잇 또는 문서에 작성

**카테고리**:

- **프로세스**: 잘 작동한 워크플로우, 도구, 방법론
- **기술**: 성공적인 기술 선택, 아키텍처 결정
- **협업**: 효과적인 의사소통, 페어 프로그래밍 등
- **개인**: 개인적 성장, 새로운 기술 습득

**예시**:

```
✅ Monorepo 도입으로 코드 공유가 원활해짐
✅ @The-Reviewer 페르소나 활용으로 코드 품질 향상
✅ 주간 1회 페어 프로그래밍으로 지식 공유 증가
```

---

#### Step 2.3: 개선할 점 수집 (20분)

**카테고리**:

- **Pain Points**: 반복되는 문제, 불편한 점
- **Blockers**: 작업을 지연시킨 요인
- **기술 부채**: 누적되고 있는 문제들
- **프로세스**: 비효율적인 절차

**예시**:

```
❌ 배포 시간이 평균 10분으로 너무 길음
❌ 테스트 커버리지 60%로 낮음
❌ API 문서가 코드와 동기화되지 않음
❌ 코드 리뷰가 24시간 이상 걸리는 경우 많음
```

**5 Whys 기법 적용**:

```
문제: 배포 시간이 10분으로 길다
Why 1: 빌드 시간이 8분 걸림
Why 2: 모든 패키지를 매번 빌드함
Why 3: 증분 빌드가 설정되지 않음
Why 4: Turborepo 캐시 설정을 안 함
근본 원인: 초기 설정 시 성능 최적화 고려 안 함
```

---

#### Step 2.4: 아이디어 도출 (20분)

**브레인스토밍**:

- 개선할 점에 대한 해결책
- 새로운 시도할 만한 것들
- 실험적 프로세스

**우선순위 투표**:

- 각자 3표씩 가장 중요한 아이디어에 투표
- 득표 순으로 정렬

**예시**:

```
💡 Turborepo 캐시 설정으로 빌드 시간 단축 (8표)
💡 Playwright E2E 테스트 도입 (6표)
💡 API 문서 자동 생성 (Swagger/OpenAPI) (4표)
💡 주간 Tech Talk 시작 (2표)
```

---

#### Step 2.5: 액션 아이템 정의 (15분)

**SMART 원칙 적용**:

- **S**pecific: 구체적
- **M**easurable: 측정 가능
- **A**chievable: 달성 가능
- **R**elevant: 관련성 있음
- **T**ime-bound: 기한 명시

**템플릿**:

```markdown
## 액션 아이템

### 1. Turborepo 캐시 설정

- **담당자**: @Architect
- **목표**: 빌드 시간을 10분 → 3분으로 단축
- **기한**: 2026-03-07
- **검증 방법**: GitHub Actions 빌드 시간 측정
- **우선순위**: High
```

---

#### Step 2.6: 체크아웃 (5분)

**회고에 대한 회고**:

- "오늘 회고는 어땠나요? (1-10)"
- "다음 회고에서 개선할 점은?"

---

### Phase 3: 문서화 및 후속 조치 (Post-Meeting)

#### Step 3.1: 회고록 작성

**`.agent/memory/RETRO_2026-02.md` 완성**:

- 모든 내용을 정리하여 문서화
- 사진 또는 스크린샷 첨부 (선택)

**공유**:

- Team Wiki 또는 Notion에 업로드
- 팀 채팅에 요약 공유

---

#### Step 3.2: Doctrine 업데이트

**Doctrine**: 팀의 원칙과 규칙을 문서화

**업데이트할 문서들**:

1. **워크플로우 개선**:
   - 비효율적인 워크플로우 수정
   - 새로운 워크플로우 추가

2. **규칙 변경**:
   - `.agent/rules/CODING_STANDARDS.md` 업데이트
   - 새로운 Best Practice 추가

3. **도구 추가**:
   - `.agent/rules/TECH_STACK.md`에 새 도구 기록

**예시**:

```markdown
## 변경 이력

### 2026-02-28: Turborepo 캐시 활성화

**배경**: 빌드 시간 단축 필요  
**변경**: `turbo.json`에 캐시 설정 추가  
**영향**: 빌드 시간 70% 감소 예상
```

---

#### Step 3.3: 액션 아이템 추적

**GitHub Issues 생성**:

```bash
gh issue create \
  --title "[RETRO] Turborepo 캐시 설정" \
  --body "회고 액션 아이템: 빌드 시간 단축을 위한 캐시 설정" \
  --label "retrospective,priority:high" \
  --assignee "@Architect"
```

**진행 상황 모니터링**:

- 다음 회고 시작 시 이전 액션 아이템 완료 여부 확인
- 미완료 항목은 이유 분석 및 연장 여부 결정

---

### Phase 4: 진화 (Evolution)

#### Step 4.1: 메타 개선 (회고의 회고)

**분기별로 회고 프로세스 자체를 평가**:

**질문**:

- 회고가 실제 개선으로 이어지고 있는가?
- 액션 아이템 완료율은? (목표: 80% 이상)
- 회고 시간이 적절한가?

**개선 예시**:

```
발견: 액션 아이템 완료율 50%로 낮음
원인: 담당자가 너무 많은 아이템을 맡음
개선: 다음 회고부터 1인당 최대 2개로 제한
```

---

#### Step 4.2: 실험 및 피드백

**새로운 프로세스는 실험으로 시작**:

```markdown
## 실험: 주간 Tech Talk

**가설**: 지식 공유를 통해 팀 전체 기술 수준 향상
**기간**: 1개월 (4회)
**측정 지표**:

- 참석률 > 80%
- 만족도 > 7/10
- 새로운 기술 적용 사례 > 2개

**결과** (1개월 후):

- 참석률: 90%
- 만족도: 8.5/10
- 적용 사례: 3개 (Zod, Tanstack Query, Playwright)
- **결론**: 정규 프로세스로 채택 ✅
```

## 🎓 회고 포맷 바리에이션

### A. Start-Stop-Continue

```
🟢 Start (시작할 것):
- Playwright E2E 테스트 도입
- 주간 Code Review 세션

🔴 Stop (중단할 것):
- 긴급하지 않은 Slack 알림
- 코드 리뷰 없는 머지

🔵 Continue (계속할 것):
- 페어 프로그래밍
- 월간 회고
```

---

### B. Sailboat Retrospective

```
⛵ 목표 (Goal): 우리가 향하는 곳
🌊 바람 (Wind): 우리를 밀어주는 것
⚓ 닻 (Anchor): 우리를 붙잡는 것
🪨 암초 (Rocks): 앞으로의 위험 요소
```

---

### C. 4Ls

```
💙 Loved (좋았던 것)
📚 Learned (배운 것)
😔 Lacked (부족했던 것)
🔮 Longed For (바라는 것)
```

## 💡 회고 Best Practices

### DO ✅

1. **심리적 안전감 조성**
   - 비난하지 않기
   - 모든 의견 존중하기

2. **구체적인 예시 사용**
   - "코드 리뷰가 늦다" (X)
   - "PR #123은 리뷰에 48시간 걸렸다" (O)

3. **액션 아이템은 SMART하게**
   - 담당자, 기한, 검증 방법 명시

4. **정기적으로 진행**
   - 월 1회 또는 스프린트마다

5. **과거 액션 아이템 리뷰**
   - 이전 회고의 개선 사항 확인

### DON'T ❌

1. **개인 공격 금지**
   - "A님이 코드를 잘못 짰다" (X)
   - "해당 모듈의 테스트 커버리지가 낮다" (O)

2. **너무 많은 액션 아이템**
   - 최대 3-5개로 제한
   - 우선순위 명확히

3. **일방적 진행**
   - 모든 팀원이 발언 기회 가져야 함

4. **형식적인 회고**
   - 실제 개선으로 이어지지 않으면 의미 없음

## 📊 회고 효과 측정

### 분기별 평가 지표

```markdown
## 2026 Q1 회고 효과 분석

| 지표         | Q1 시작 | Q1 종료 | 개선율 |
| ------------ | ------- | ------- | ------ |
| 배포 빈도    | 주 1회  | 주 3회  | +200%  |
| 버그 발생률  | 10%     | 3%      | -70%   |
| PR 리뷰 시간 | 24시간  | 8시간   | -67%   |
| 팀 만족도    | 6.5/10  | 8.2/10  | +26%   |

**성공 요인**:

- 회고 액션 아이템 완료율 85%
- 정기적인 실험 및 피드백

**다음 분기 개선**:

- 테스트 커버리지 60% → 80% 목표
```
